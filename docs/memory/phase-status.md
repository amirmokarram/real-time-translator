- **6E DROPPED — Phase 6 CLOSED (2026-07-21).** Amir raised that Whisper isn't a good fit for this app. Agreed diagnosis: the problem is **architecture, not accuracy** — Whisper is a *batch* model (30s training windows) and WhisperLive fakes streaming with a sliding window, hence the latency floor, self-revising segments (which fight the app's commit-a-sentence-once design), and hallucination on short/silent chunks. On a *complete* utterance `large-v3` is excellent, so it would be a strong choice if file/recording transcription were ever added. **Decision: keep Whisper, stop investing in it.** Kept because it's the only offline path (confidential/NDA meetings can't stream to a cloud vendor), is cleanly isolated behind `ISttStream` (costs nothing to keep), and is a fallback for an expired key/outage — removing it would be real work with a downside and zero gain for daily use. Instead of building 6E: **honest labeling** — Settings copy now recommends DeepGram for live and frames Whisper as the offline/private option with higher latency + self-revising output (engine dropdown labels + a Whisper-only hint noting the accuracy knobs are DeepGram-only); README updated to match. **Do not re-propose Whisper accuracy work** unless offline mode becomes a real use case or file transcription gets added.

**DONE — Phase 7/8/9: Meeting recording, review & notes (2026-07-21, built in one session; NOT yet live-tested by Amir).** Motivation in his words: "after live translation I can save all translated content, now I want to record all voices because after the meeting I want to review it and take note." Text alone is lossy — a low-confidence row says the recognizer struggled but not what was really said. Five phases, each approved in turn:

- **7A — record the session.** New `RecordingService` (`src/app/core/services/recording.service.ts`) owns its **own MediaRecorder**, deliberately NOT the STT one: `DeepGramStream` rebuilds its recorder on every reconnect, so a shared recording would have holes. The recorded mix **never reaches the recognizer** — mixing the mic in is recording-only, so Amir's own voice is never transcribed and translated back at him. Mixer uses **its own AudioContext at the default sample rate** (AudioService's is pinned to 16 kHz for the level meter; reusing it would render a 16 kHz recording). Mic modes only engage when capturing **system audio** (capturing a mic already records that voice). Chunks stream to main via IPC every 5 s → `electron/recording-store.ts` write streams (crash costs ≤5 s, not the meeting). Settings → **Recording** panel (own `SettingsNode` leaf): enabled · folderPath (''=`userData/recordings`) · mode (`source`/`mix`/`separate`) · micDeviceId · micGain (live GainNode) · bitrateKbps (default 64, vs 32 for STT — one is for a recognizer, one for ears). `● REC 12:04` badge in the audio bar. **Recording failure never stops translation.**
- **7B — transcript sidecar.** `meeting-….json` beside the audio: `{startedAt, durationMs, languages, entries:[{offsetMs, source, target, provider, confidence}]}`. **The timestamp that matters:** `TranslationEntry.timestamp` is when the *translation returned*, 1–3 s late — seeking to it lands past the line. Added `bufferStartedAt` to `TranscriptionService` (mirrors `bufferConfidence` exactly) → `PendingSentence.startedAt` → `TranslationEntry.startedAt`. SRT export now zeroes on the **recording start** (not the first row) so subtitles drop onto the audio. Typed rows are excluded from the sidecar — no place in the audio.
- **7C — Review view.** Lazy route `#/review` + header nav. Session list (date/duration/lines/size) + player + transcript; click a line to seek, playing line highlights. **Playback over a custom `rec://` scheme, not `file://`:** the dev CSP (`default-src 'self'`) refuses file:// media, and registering the scheme with `stream: true` + `net.fetch` gives **range requests**, which is what makes seeking in a 50 MB file instant. Handler takes a **basename only** through `resolveSessionFile` (path.basename + inside-folder check) since the renderer supplies it. **Scrubber reads `durationMs` from the sidecar, never `audio.duration`** — a streamed WebM has no duration header and reports `Infinity`. Delete uses `shell.trashItem` (recoverable), not unlink.
- **8 — Notes.** Session-level note + per-line notes in Review; 700 ms debounce plus explicit flush on blur/session-switch/teardown. Stored **in the same sidecar** (one self-contained file per session). **Main does the merge** (`saveNotes` replaces only the `notes` key) — if the renderer posted back a whole transcript it read earlier, a stale copy would silently overwrite the real one. Writes go **temp file + rename** so an interrupted save can't corrupt the transcript. Line notes keyed by **offsetMs**, not index — the offset *is* what the note is about.
- **9 — Assist on past sessions.** Player-bar **Ask** (whole session transcript) + per-line Ask, both via `AssistService.openWith()` sending **source lines only** (same shape as the translator's selection→Ask). Long meetings are sent **whole, not truncated** (cutting either end loses content silently; a 90-min meeting ≈20k tokens fits cloud providers) — above ~40k chars the button shows a warning badge, since a small-context local Ollama model would quietly see only part. **DONE — Five defects found after the first live test (2026-07-21, same day).** Amir reported "text is not synced with audio" in Review, then "clicking a row resets the audio and jumps to start". Both were real; three more surfaced while fixing them. Diagnosis was driven by **evidence, not guesses** — reading the timing numbers out of a real sidecar (offsets only, not his meeting text) and measuring the media element's state in a throwaway spec.

1. **Sentences shared one timestamp** (the visible desync). A real sidecar showed offsets `9738, 9738, 9738, 9738, 23812, 23812, …`. First attempt was per-fragment marks, but that was still insufficient — **a single finalized fragment routinely contains several sentences** (e.g. "One. Two. Three." in one Deepgram `is_final`), so they still shared the fragment's start. **Final fix: per-WORD timing.** `SttCallbacks.final` takes an `SttFinal` object (was 4 positional args, now `{text, endOfUtterance, confidence, startedAt, wordStartedAt}`); Deepgram maps `alternatives[0].words[].start` → wall-clock via `streamStartWall`. `TranscriptionService` keeps a `pendingWords: number[]` (one start-time per word in the buffer); each committed sentence takes its first word's time and splices off `countWords(sentence)` entries. Whisper/mock report no word times → fall back to one time for the whole fragment. This is what actually removes the duplicate offsets.
2. **Timestamps were arrival times, not speech times.** Even one sentence per buffer was ~1–2 s late (his `endpointingMs` is 1000, so a fragment finalizes a second after the pause). **DeepGram already sends the answer and we were discarding it:** Results messages carry `start`/`duration` and `words[]` with per-word `start`/`end`, in seconds from the start of that connection's audio. `DeepGramStream` now stamps `streamStartWall` at recorder start (**reset on every reconnect** — each connection counts from 0) and converts `words[0].start` into a wall-clock moment, passed as `SttCallbacks.final(..., speechStartAt)`. Whisper reports no timings → falls back to arrival time.
3. **`rec://` didn't serve byte ranges** (the "jumps to start" bug). Delegating to `net.fetch` on a `file://` URL produced no usable range response. Measured on a 15 s recording: buffered **0–2.34 s only**, and seeking to 12 s **landed at 0**; after the fix it lands at 12. A streamed WebM has no duration header, so **range support is the only thing making it seekable**. The handler now parses `Range` itself and returns `206` + `Content-Range`/`Content-Length`/`Accept-Ranges`/`Content-Type`, handling `bytes=N-M`, `bytes=N-`, `bytes=-N` and `416`. Plus **duration priming**: one forced seek past the end on load makes Chromium learn the real length (it reports `Infinity` otherwise); a click arriving mid-prime is held and applied after.
4. **Navigating away mid-capture lost the transcript.** `ngOnDestroy` stopped capture without awaiting, and the isCapturing effect died with the component → audio but no sidecar. Now chains the finish onto `stopCapture()`; `finishSession()` is re-entrant (returns the in-flight promise) since the effect and ngOnDestroy can both reach it.
5. **`separate` mode listed a phantom session** — `listSessions` paired every `.webm` with a same-named `.json`, so `-mic.webm` appeared as its own transcript-less session. Mic files now attach to their `-system` sibling as `micFile`, Review shows a **System | Mic** toggle (same timeline, so offsets stay valid), and deleting takes both.
6. **Notes could overwrite each other** — `flushNotes` posted the whole notes object without serializing, so blurring the session note and a line note in quick succession could write the line note then clobber it with the earlier state. Now serialized through a chain, same pattern as the recording `writeChain`.
8. **Notes lost intermittently on Windows** (found by `--repeat-each 12` on a "flaky" test): the sidecar's atomic temp→rename fails with `EPERM` when anything else holds the file open, ~30% of runs. Now retries with backoff and falls back to an in-place write. See [[gotchas-and-lessons]].
9. **Review showed whole seconds**, so rows a few hundred ms apart looked identical even after the offsets were fixed — Amir could see the JSON was right but the UI wasn't. Transcript rows now use `rowStamp()` (`m:ss.mmm`); the player bar keeps `m:ss`.
7. **Line notes keyed by offsetMs, which is not unique** (found after the word-timing fix — same duplicate-timestamp root cause). When two rows shared an offset, clicking one row's note button opened BOTH editors and a note applied to both. Line notes now key by transcript **row index** in the Review UI; the sidecar `notes.lines[]` gained `index` as identity while keeping `offsetMs` for readability, and old files (offset-only) are matched back by `findIndex(offsetMs)` on load. **Lesson: an audio offset is a fine thing to display, but a terrible key — several sentences can start at the same instant.**

**Amir has NOT fully live-tested this yet.** Three things the silent fake-audio E2E device cannot prove: the **mic-mixing modes** (need a real system-audio session), whether **offsetMs lands on the right words**, and **seeking in a long cue-less WebM** (Chromium has to scan clusters; there is a standard duration-priming workaround deliberately left out rather than adding speculative complexity).

**DONE — Commit a sentence the moment its punctuation is unambiguous (2026-07-22, Amir tested live: "It seems better than previous version").** He asked why he waits for a translation when he can already see a full stop or question mark on screen. Two separate waits, and the answer named both: (1) the punctuation he sees is often on an **interim** hypothesis — `smart_format` punctuates those too, and nothing enters segmentation until `is_final`, deliberately (translating a hypothesis means visibly rewriting rows when it changes); (2) `drainSentences` only peeled a sentence once the NEXT one started arriving (`\s+(?=\S)`), so the trailing — freshest — sentence paid the whole endpointing silence (~800ms–1s) before reaching the translator. **Fix:** `commitCompleteTail()` runs after every finalized fragment and commits the tail immediately when it ends in unambiguous `.!?`. The abbreviation guard had to be shared with `drainSentences` to be worth anything — guarding only the tail would hold "I met Dr." and then split it identically one fragment later, i.e. the same output, just slower. So the anchored `sentenceRe` became a **global `boundaryRe` + `findSentenceEnd()`** walking candidate breaks, with one `isRealBoundary()` predicate vetoing abbreviations and dotted initialisms (`e.g.`, `U.S.`, `J.`). Net effect beyond latency: **"I met Dr. Smith yesterday." is now one row**, and the fragment "I met Dr." is never sent to a translator as if it were a sentence. **Deliberately NO decimal rule** — see [[gotchas-and-lessons]]. Abbreviation list deliberately omits `no.`/`al.` (collide with the word "no" and the name "Al"; a false positive glues two real sentences, which is worse than one wrong split). `endUtterance` still commits a punctuated tail **without** vetting: real silence outweighs what the punctuation looks like. E2E: 3 tests under `trailing-sentence commit` seeded `sentenceMaxWaitMs: 60000` so only the punctuation rules can satisfy them, never the idle timer firing behind the assertion (new `SeedOverrides.sentenceMaxWaitMs`).
