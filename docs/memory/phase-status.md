- **7A ‚Äî record the session.** New `RecordingService` (`src/app/core/services/recording.service.ts`) owns its **own MediaRecorder**, deliberately NOT the STT one: `DeepGramStream` rebuilds its recorder on every reconnect, so a shared recording would have holes. The recorded mix **never reaches the recognizer** ‚Äî mixing the mic in is recording-only, so Amir's own voice is never transcribed and translated back at him. Mixer uses **its own AudioContext at the default sample rate** (AudioService's is pinned to 16 kHz for the level meter; reusing it would render a 16 kHz recording). Mic modes only engage when capturing **system audio** (capturing a mic already records that voice). Chunks stream to main via IPC every 5 s ‚Üí `electron/recording-store.ts` write streams (crash costs ‚â§5 s, not the meeting). Settings ‚Üí **Recording** panel (own `SettingsNode` leaf): enabled ¬∑ folderPath (''=`userData/recordings`) ¬∑ mode (`source`/`mix`/`separate`) ¬∑ micDeviceId ¬∑ micGain (live GainNode) ¬∑ bitrateKbps (default 64, vs 32 for STT ‚Äî one is for a recognizer, one for ears). `‚óè REC 12:04` badge in the audio bar. **Recording failure never stops translation.**
- **7B ‚Äî transcript sidecar.** `meeting-‚Ä¶.json` beside the audio: `{startedAt, durationMs, languages, entries:[{offsetMs, source, target, provider, confidence}]}`. **The timestamp that matters:** `TranslationEntry.timestamp` is when the *translation returned*, 1‚Äì3 s late ‚Äî seeking to it lands past the line. Added `bufferStartedAt` to `TranscriptionService` (mirrors `bufferConfidence` exactly) ‚Üí `PendingSentence.startedAt` ‚Üí `TranslationEntry.startedAt`. SRT export now zeroes on the **recording start** (not the first row) so subtitles drop onto the audio. Typed rows are excluded from the sidecar ‚Äî no place in the audio.
- **7C ‚Äî Review view.** Lazy route `#/review` + header nav. Session list (date/duration/lines/size) + player + transcript; click a line to seek, playing line highlights. **Playback over a custom `rec://` scheme, not `file://`:** the dev CSP (`default-src 'self'`) refuses file:// media, and registering the scheme with `stream: true` + `net.fetch` gives **range requests**, which is what makes seeking in a 50 MB file instant. Handler takes a **basename only** through `resolveSessionFile` (path.basename + inside-folder check) since the renderer supplies it. **Scrubber reads `durationMs` from the sidecar, never `audio.duration`** ‚Äî a streamed WebM has no duration header and reports `Infinity`. Delete uses `shell.trashItem` (recoverable), not unlink.
- **8 ‚Äî Notes.** Session-level note + per-line notes in Review; 700 ms debounce plus explicit flush on blur/session-switch/teardown. Stored **in the same sidecar** (one self-contained file per session). **Main does the merge** (`saveNotes` replaces only the `notes` key) ‚Äî if the renderer posted back a whole transcript it read earlier, a stale copy would silently overwrite the real one. Writes go **temp file + rename** so an interrupted save can't corrupt the transcript. Line notes keyed by **offsetMs**, not index ‚Äî the offset *is* what the note is about.
- **9 ‚Äî Assist on past sessions.** Player-bar **Ask** (whole session transcript) + per-line Ask, both via `AssistService.openWith()` sending **source lines only** (same shape as the translator's selection‚ÜíAsk). Long meetings are sent **whole, not truncated** (cutting either end loses content silently; a 90-min meeting ‚âà20k tokens fits cloud providers) ‚Äî above ~40k chars the button shows a warning badge, since a small-context local Ollama model would quietly see only part. Assist context chip renamed "Context from selected rows" ‚Üí "**Transcript context**" (three entry points now fill it).

Two bugs the tests caught, both real: (1) chunk writes were **fire-and-forget**, so `recordingStop` could close the file before the last chunk landed *and* two overlapping `blob.arrayBuffer()` reads could interleave into a corrupt WebM ‚Üí fixed with a serialized `writeChain` drained on stop; (2) the **tray menu / global hotkey** stop path (`CommandService`) never finished the session, so those recordings would have had no sidecar ‚Üí the translator now watches `audio.isCapturing()` for the true‚Üífalse edge, covering every stop path once (note left in `command.service.ts` so nobody "fixes" it by duplicating). Also: `lastSession` is published **synchronously** at the top of `RecordingService.stop()`, before any await, or watchers race it.

**DONE ‚Äî Five defects found after the first live test (2026-07-21, same day).** Amir reported "text is not synced with audio" in Review, then "clicking a row resets the audio and jumps to start". Both were real; three more surfaced while fixing them. Diagnosis was driven by **evidence, not guesses** ‚Äî reading the timing numbers out of a real sidecar (offsets only, not his meeting text) and measuring the media element's state in a throwaway spec.

1. **Sentences shared one timestamp** (the visible desync). A real sidecar showed offsets `9738, 9738, 9738, 9738, 23812, 23812, ‚Ä¶`. `bufferStartedAt` was set once when the buffer filled from empty and cleared only when it fully drained ‚Äî but one buffer routinely yields several sentences, so they all inherited the FIRST fragment's time. Replaced with **per-fragment marks** (`{index, at}` sliding as text is consumed): each sentence takes the time of the fragment it actually starts in. See `consumeMarks()` in `transcription.service.ts`.
2. **Timestamps were arrival times, not speech times.** Even one sentence per buffer was ~1‚Äì2 s late (his `endpointingMs` is 1000, so a fragment finalizes a second after the pause). **DeepGram already sends the answer and we were discarding it:** Results messages carry `start`/`duration` and `words[]` with per-word `start`/`end`, in seconds from the start of that connection's audio. `DeepGramStream` now stamps `streamStartWall` at recorder start (**reset on every reconnect** ‚Äî each connection counts from 0) and converts `words[0].start` into a wall-clock moment, passed as `SttCallbacks.final(..., speechStartAt)`. Whisper reports no timings ‚Üí falls back to arrival time.
3. **`rec://` didn't serve byte ranges** (the "jumps to start" bug). Delegating to `net.fetch` on a `file://` URL produced no usable range response. Measured on a 15 s recording: buffered **0‚Äì2.34 s only**, and seeking to 12 s **landed at 0**; after the fix it lands at 12. A streamed WebM has no duration header, so **range support is the only thing making it seekable**. The handler now parses `Range` itself and returns `206` + `Content-Range`/`Content-Length`/`Accept-Ranges`/`Content-Type`, handling `bytes=N-M`, `bytes=N-`, `bytes=-N` and `416`. Plus **duration priming**: one forced seek past the end on load makes Chromium learn the real length (it reports `Infinity` otherwise); a click arriving mid-prime is held and applied after.
4. **Navigating away mid-capture lost the transcript.** `ngOnDestroy` stopped capture without awaiting, and the isCapturing effect died with the component ‚Üí audio but no sidecar. Now chains the finish onto `stopCapture()`; `finishSession()` is re-entrant (returns the in-flight promise) since the effect and ngOnDestroy can both reach it.
5. **`separate` mode listed a phantom session** ‚Äî `listSessions` paired every `.webm` with a same-named `.json`, so `-mic.webm` appeared as its own transcript-less session. Mic files now attach to their `-system` sibling as `micFile`, Review shows a **System | Mic** toggle (same timeline, so offsets stay valid), and deleting takes both.
6. **Notes could overwrite each other** ‚Äî `flushNotes` posted the whole notes object without serializing, so blurring the session note and a line note in quick succession could write the line note then clobber it with the earlier state. Now serialized through a chain, same pattern as the recording `writeChain`.

**Amir has NOT fully live-tested this yet.** Three things the silent fake-audio E2E device cannot prove: the **mic-mixing modes** (need a real system-audio session), whether **offsetMs lands on the right words**, and **seeking in a long cue-less WebM** (Chromium has to scan clusters; there is a standard duration-priming workaround deliberately left out rather than adding speculative complexity).
